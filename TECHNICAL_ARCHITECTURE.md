## 1. What problem the pipeline solves

Wolters Kluwer's KLI (Kluwer Law International) repositories — Kluwer Arbitration (KA), Kluwer Competition Law (KCL), KIPL, and others — store their content as XML conforming to the KEA-BASIC DTD. Each document carries some editorial enrichment (`<topic>`, `<keywords>`, `<organization>`, `<country>`) but coverage is uneven and the existing tags are too sparse to power modern AI-search, semantic retrieval, or knowledge-graph products.

**Goal of this pipeline:** read every KEA-BASIC XML document, restore it into a clean structured form, then enrich it with a 4-level taxonomy (L1 Domain → L2 Macro-Topic → L3 Sub-Topic → L4 Entities + Keywords) using LLM and embedding techniques. Output is RAG-ready chunks plus a document-level rollup ready to feed into a search index, an AI assistant, or a knowledge graph.

**Demonstrated on:** 23 KA + 21 KCL = 44 source XML files, 974 retrieval chunks, end-to-end on Gemini 2.5 Flash + `gemini-embedding-001`.

## 2. What's wrong with the raw XML

Three issues actively crash naïve parsers; the pipeline detects and pre-processes around all three before XML parsing begins.

| Problem | Where | Frequency in sample | Symptom |
|---|---|---|---|
| `<!DOCTYPE … [ <!ENTITY SLSD_…_gif … NDATA GIF> … ]>` (internal subset declaring binary entities) | DOCTYPE | 1/23 KA (4%), 5/21 KCL (24%) | `xml.etree` rejects the file outright |
| Legacy commented-out `<frontmatter>` blocks | inside `<KEA-publ>` | 13/23 KA (56%), 18/21 KCL (86%) | Naïve parsers see two frontmatter blocks → duplicated authors / dates |
| `<?page nr="…"?>` processing instructions | inline inside `<p>` tags | every paginated doc | Whitespace duplication on text flatten |

In addition, the editorial content is **inverted between corpora**:
- KA has 22% keyword coverage but only **4%** topic coverage
- KCL has 4% keyword coverage but **57%** topic coverage

So the pipeline must treat both layers as first-class — neither editorial topics nor editorial keywords can be assumed populated.

`src.rag_parser.preprocess()` strips the three landmines in a single bytes-level pass before any XML parsing, ensuring 100% parse success. Documented in detail in [data/ka_eda/FINDINGS.md](data/ka_eda/FINDINGS.md) and [data/kcl_eda/FINDINGS.md](data/kcl_eda/FINDINGS.md).

## 3. The 4-level taxonomy

The conceptual core. Every chunk and every document is tagged at four hierarchy levels, each of which is generated by a different mechanism:

| Level | Name | Source | Example (KA) | Example (KCL) |
|---|---|---|---|---|
| **L1** | Domain | Deterministic, from `cust_groups` | `International Arbitration` | `Competition Law` |
| **L2** | Macro-Topic | Anchor-or-Expand routing (cosine + Gemini) | `State Immunity` | `Merger Control` |
| **L3** | Sub-Topic | Unsupervised clustering then Gemini-named | `ICSID Waiver of Immunity` | `Combination Notification Requirements` |
| **L4** | Entities + Keywords | Gemini structured-output extraction | `Halliburton v. Chubb`; `Section 9 SIA` | `Article 22 EUMR`; `Illumina/GRAIL` |

**Why four levels and not, say, just topics + keywords?**
- L1 is the routing key for product-line filtering ("show me arbitration content only")
- L2 is the broad subject for autosuggest and assistant grounding ("about Mergers")
- L3 is the discoverable angle within a subject ("specifically Combination Notification Requirements") — emergent from data, not pre-declared
- L4 is the precision layer for highlighting, citation linking, and entity-aware search ("show me documents that cite Article 22 EUMR")

Each level is tracked with its source-of-truth on the chunk (e.g. `L2_Source = "anchor"|"generator"`, `L2_Similarity = 0.785`), so the document-level rollup can prove which decisions were cheap dictionary hits and which used an LLM.

## 4. Phase-by-phase pipeline

```
                                 ┌─────────────────────────┐
                                 │  KA-xml-samples/*.xml   │
                                 │  KCL-xml-samples/*.xml  │
                                 └────────────┬────────────┘
                                              ▼
PHASE 1 — Parse XML to JSON                   src/rag_parser.py
  Strip DOCTYPE+entity-subsets, comments, page PIs
  Normalise 5 doc-types into a single output schema
  Output: ka_parsed.json, kcl_parsed.json
                                              ▼
PHASE 2 — Chunk + Dictionary + Graph
  src/semantic_chunker.py   Paragraph Group Chunking (target 500w,
                            slack 150, hard cap 800; never mid-paragraph)
                            + Prefix-Fusion lineage breadcrumb
  src/tag_harvester.py      Domain-partitioned master_dictionary.json
                            (seed merge / pristine modes)
  src/relationship_extractor.py  xref → directed edges
                                              ▼
PHASE 3 — Enrichment (L1 + L2 + L3 + L4)      src/phase3/*
  3.1 Router           (router.py)        L1 + L2 anchor-or-expand
  3.2 L4 Extractor     (l4_extractor.py)  Cases, statutes, orgs, kws
  3.3 L3 Discovery     (l3_discovery.py)  Cluster-then-Label per L2
  Pluggable providers in providers.py:
      Embedder, TopicGenerator, EntityExtractor, SubTopicNamer
      Concrete: Gemini API, Vertex AI, Sentence-Transformers, Dummy
                                              ▼
PHASE 4 — Document synthesis                  src/phase4/synthesize.py
  Group chunks by doc_id; word-weighted L2 ranking;
  case-fold deduped L4 entities; L2_Provenance summary;
  Before/After --compare visualizer
  Output: final_enriched_documents.json + compare_all.txt

                          (POST-RUN)          src/phase4/expand_dictionary.py
  Harvest all L2 / L3 / L4 from enriched chunks
  Produce enriched_dictionary.json with seed vs discovered split
  + frequency counts for every term
```

Each module is independently runnable, has its own CLI, and writes a single canonical JSON artefact. Downstream phases consume those artefacts. **No hidden cross-module state.** This makes the pipeline:
- *resumable* — kill mid-run and restart, lost work is bounded by the last-completed phase
- *parallelisable* — chunks are independent within a phase
- *auditable* — every intermediate is on disk in human-readable JSON

## 5. Anchor / Expand routing gate

This is the main cost-saving design and the source of the "85% anchored" demo headline.

### Algorithm (src/phase3/router.py)

For each chunk:

1. Embed `fused_text` once with the configured embedder (`gemini-embedding-001` → 3072-d, or `bge-small` → 384-d).
2. Embed all L2 anchor topics from `master_dictionary[L1].topics` once at TopicRouter initialisation; cache.
3. Compute max cosine similarity between chunk vector and anchor matrix.
4. **If `max_sim >= recommended_anchor_threshold`** → **ANCHOR**: assign the matched anchor's L2 verbatim. **No generator call.**
5. **Else** → **EXPAND**: call Gemini 2.5 Flash with the spec prompt + strict JSON schema. Get back `{proposed_l2_topic, reasoning}`.

### Self-calibrated threshold

Each embedder class advertises `recommended_anchor_threshold`:

| Embedder | Threshold | Rationale |
|---|---|---|
| `SentenceTransformerEmbedder` (bge-small) | 0.85 | bge cosines for related legal text peak around 0.88+ |
| `GeminiEmbedder` (gemini-embedding-001) | 0.75 | Gemini cosines peak around 0.82-0.84; 0.85 would never anchor |
| `DummyEmbedder` | 0.30 | hashing-based; near-orthogonal vectors |

`TopicRouter._auto_threshold(embedder)` looks up the class attribute. **Override** via `anchor_threshold=` (library) or `--threshold` (CLI). No magic numbers in calling code.

### Why the L2 dictionary uses pristine seeds

Two modes for `master_dictionary.json`:
- **Merge mode** (default): seed taxonomy + editorial XML topics coexist
- **Pristine mode** (`--drop-harvested-topics`): seeds only

The 4-doc demo used pristine; the full 44-doc POC used **merge** mode (you asked for "all topics from all docs"). Pristine produces cleaner anchor decisions; merge produces a richer harvested vocabulary. Real production would likely be pristine — the "noise" topics from XML editorial tags compete with curated seed topics for anchor matches.

## 6. Cluster-then-Label for L3

Asking an LLM to invent L3 sub-topics directly is expensive and inconsistent across batches (the same chunk gets a different L3 every time). Cluster-then-Label flips it: cluster first, then ask the LLM to *name* the cluster after seeing what's in it.

### Algorithm (src/phase3/l3_discovery.py)

For each L2 group:

1. **Embed** all member chunks (re-embeds, doesn't reuse 3.1's vectors — those weren't persisted).
2. **Cluster** with `AgglomerativeClustering(metric="cosine", linkage="average", distance_threshold=0.35)`.
3. **Synthetic noise bucket**: any cluster with size < `min_cluster_size` (default 2) is collapsed to label `-1`. This mirrors HDBSCAN's noise handling without requiring the extra dependency.
4. **Centroid pick**: for each real cluster, compute the mean vector and select the K=3-5 chunks whose vectors are closest (cosine) to it.
5. **Label generation**: concatenate centroid texts (with a `[Chunk <id>]` separator) and call Gemini with the L3 prompt. Get back `{l3_sub_topic, justification}`.
6. Apply the L3 label to every chunk in the cluster. Noise chunks get `General {L2_Topic}`.

### Why cosine + agglomerative + threshold (vs k-means)

- **Cosine** matches the embedding model's training objective; k-means' Euclidean does not
- **Threshold-based** doesn't require pre-declaring K — clusters of arbitrary natural size form
- **Agglomerative** is deterministic given the same input (k-means is initialisation-sensitive)
- Single sklearn dependency, no HDBSCAN install

### Critical bug fix

The L3 merge step originally keyed labelled chunks by `chunk_id` alone. Chunk IDs are **doc-local** (`a0001_chunk_1` appears in many different docs), so name-collisions corrupted ~40% of chunks in the first full run. Fixed in commit by keying on `(doc_id, chunk_id)`. Anyone reusing `discover_l3_for_l2` in a custom driver should use the same compound key.

## 7. L4 entity & keyword extraction

Gemini 2.5 Flash with **structured JSON output** (response_mime_type + response_schema). Per chunk:

```
{
  "entities": {
    "case_names":               [...],
    "statutes_and_regulations": [...],
    "organizations":            [...]
  },
  "keywords": {
    "existing_matched_keywords": [...],   // FROM the per-domain dictionary
    "new_extracted_keywords":    [...]    // novel concepts
  }
}
```

The prompt sends the per-domain seed keyword list and asks Gemini to **echo verbatim** when a chunk concept matches a dictionary keyword. This is not enforceable — Gemini occasionally paraphrases.

### Defensive validation

`l4_extractor._validate_matched_keywords()` post-checks every string in `existing_matched_keywords` against the actual dictionary set. Anything not literally in the dictionary is **demoted** into `new_extracted_keywords`. The controlled-vocabulary guarantee does not depend on the model obeying instructions.

### Failure recovery

- Transient 429 / 503: provider-level exponential backoff (4 retries).
- Persistent parse failure on a single chunk: record `L4_metadata=null`, don't abort the doc.
- Server-side gRPC cancellation (rare): same null-record path, surfaces in audit summary.

In our 974-chunk run, **973/974** chunks were L4-extracted successfully. The single failure was a server-side `RPC::CANCELLED` on Gemini's end on an oversized chunk; we left it null rather than retry indefinitely.

## 8. Document synthesis (rollup logic)

`src/phase4/synthesize.py` groups chunks by `doc_id` and produces a per-document record. Key design choices:

### Word-weighted L2 ranking (production refinement)

Naïve "primary L2 = most chunks" is wrong because chunks vary in length 14-800 words. A 40-word aside on Cartels shouldn't outweigh an 800-word section on Merger Control.

Implementation:
- For each chunk, count body words *after* stripping the Prefix-Fusion breadcrumb (so boilerplate doesn't inflate weights).
- Accumulate per-L2 `word_weight` Counter alongside `chunk_count`.
- `L2_Primary_Topics` ranked: word_weight desc → chunk_count desc → name asc (deterministic).
- Threshold for "primary": leader_weight × 0.5. Allows up to 2 primaries per spec.

### Case-folded entity dedup

LLM occasionally returns the same entity in different casing (`European Commission` vs `european commission`). `_casefold_dedup()` groups by `casefold()` key, picks the representative with the best title-case score (custom heuristic — first letter capital, stopwords like *of*/*the* lowercase, all-caps acronyms preserved). Tested on `Singapore Court of Appeal` vs `Singapore Court Of Appeal` — picks the editorial-correct lowercase-`of` variant.

### L2_Provenance summary

For every doc, count how many chunks anchored vs expanded:

```
"L2_Provenance": {
  "summary":        "Mixed: 75% Anchored, 25% Expanded",
  "anchored_count": 18,
  "expanded_count": 6,
  "total":          24,
  "anchored_pct":   75.0,
  "expanded_pct":   25.0
}
```

This is the cost story made concrete: **for documents marked 100% Anchored, the pipeline spent zero LLM-generation calls on L2** — only embedding + cosine.

## 11. Cost & quota model

Gemini calls are the only paid resource. Per-chunk:

| Phase | Calls per chunk | Calls in full POC (974 chunks) |
|---|---|---:|
| 3.1 embedding | 1 (gemini-embedding-001) | 974 |
| 3.1 L2 generation (Expand only) | 0 if anchored, 1 if expanded | 150 |
| 3.2 L4 extraction | 1 (gemini-2.5-flash) | 974 |
| 3.3 embedding (re-embed for clustering) | 1 | 974 |
| 3.3 L3 naming (per L2 cluster, not per chunk) | — | 127 |
| **Total Gemini API calls** | | **~3,200** |

**Anchor savings:** if all 974 chunks had gone through Expand instead of Anchor at 3.1, that would be +824 generation calls (a 26% increase in total call volume). The anchor gate's value scales with the seed dictionary's coverage — at production scale with a thousand-topic seed dictionary, anchor rates likely climb above 95%.

### Free-tier reality

- `gemini-embedding-001` rate-limits aggressively (~5 RPS, daily quota cap)
- `gemini-2.5-flash` softer but still rate-limited
- Pipeline implements 4-retry exponential backoff and per-request 30-60 s timeouts
- Document-level checkpointing (per_doc/*.json) makes interruptions recoverable

For production / SLO-bound runs, switch to a paid Vertex AI project — eliminates the rate-limit churn and cuts wall-clock by 10×.

## 12. Known limitations & future work

### Active limitations

1. **L2 fragmentation at low anchor density.** When the seed dictionary is small (24 entries), Gemini's Expand path proposes slightly-different L2 names for semantically close chunks (`"Digital Markets"` vs `"Digital Markets Competition"` vs `"Digital Market Dominance"`). 127 L2 groups emerged from 974 chunks in the demo — at full corpus scale this self-corrects through anchor growth. Mitigation: promote high-frequency discovered L2s back into the next seed iteration.

2. **`google.generativeai` SDK deprecated.** Google has ended support in favour of `google.genai`. Works today but should migrate before the first non-POC release.

3. **Title-casing on case names is imperfect.** L4 occasionally returns `Djp V. Djo` instead of `DJP v. DJO` (acronym preservation in a title-case phrase). 30-min post-processing fix.

4. **Recursive section chunking not yet implemented.** Current chunker flattens past top-level sections. Most docs have nested sections (depth 3-4) that would chunk more semantically with recursive descent. A refinement, not a correctness issue.

5. **No formal pytest suite yet.** Smoke tests live inside the CLI demo runners.

### Designed-for-extension surfaces

- **More domains:** add a `KIPL` block to `seed_topics.json`, register it in `tag_harvester.DEFAULT_DOMAINS`, route accordingly.
- **Different embedders:** implement the `Embedder` protocol, advertise a `recommended_anchor_threshold` class attribute, register in `make_embedder()`.
- **Different LLM provider:** implement `TopicGenerator` / `EntityExtractor` / `SubTopicNamer` protocols.
- **Larger corpora:** doc-by-doc driver (`src/phase3/run_full_poc.py`) parallelises trivially — each doc is independent.
