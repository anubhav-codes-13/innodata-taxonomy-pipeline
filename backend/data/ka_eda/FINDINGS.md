| Inner tag | Count | What it holds | Example |
|---|---|---|---|
| `<essay>` | 14 | Journal articles (`essay-type="articles"`) and book chapters / handbook national reports (`essay-type="monograph"`) | `KLI-JOIA-420501.xml`, `KLI-KA-AI-Arbitration-2026-Ch04.xml` |

| `<booktoc>` | 1 | Table-of-contents pointing (via `<xref>`) at other KA docs | `TOC-AI-Arbitration-2026.xml` |

Everything is **English** (`lang="en"`) in this sample. Every document carries ISBN or ISSN.

### Publisher / container

### Product-tagging (`<cust-group>` — already in data!)
All 23 files carry at least `cust-group="KA"`. The second tag segments the sub-product:
- `(KA,)` — 12 docs
- `(KA, IHB)` — 6 docs (ICCA Int'l Handbook)
- `(KA, CH)` — 5 docs (journal content)

**This is ground-truth product/segment labeling the enrichment pipeline can train against or evaluate on.**

### Publication timeline
`2019: 1, 2022: 2, 2023: 6, 2025: 5, 2026: 8` — the sample is biased toward forthcoming 2026 AI-arbitration content, plus a long tail of handbook supplements.

## 2. The enrichment gap (the real finding)

The DTD provides dedicated enrichment slots: `<topic>`, `<keywords>`, `<text-scope>/<country>`, `<text-scope>/<organization>`. **They are almost entirely empty.**

| Field | Populated docs | % | Notes |
|---|---|---|---|
| `<topic>` | 1 / 23 | **4.3%** | Only `KLI-JOIA-420501` ("Investment Arbitration") |
| `<keywords>` | 5 / 23 | **21.7%** | 4 JOIA articles + 1 case-note carry 4–10 keywords each; everything else has none |
| `<country>` | 5 / 23 | 21.7% | Only ICCA national reports populate this (Latvia, etc.) |
| `<organization>` | 2 / 23 | 8.7% | Only 2 JOIA articles |
| ISBN/ISSN | 23 / 23 | 100% | |

**Breakdown by doc type:**

| doc_type | topic % | keyword % |
|---|---|---|
| essay (articles) | ~25% | 100% (4/4 JOIA articles have keywords) |
| essay (monograph) | 0% | 0% (ALL 10 book chapters + handbook reports are unenriched) |
| legislation | 0% | 0% |
| commentary | 0% | 33% |
| booktoc | 0% | 0% |

**This is the central observation:** journal articles (JOIA) already come with 4–10 keywords and a single topic. **Everything else — book chapters, handbook reports, legislation, case-notes — is a blank slate.** That's your enrichment opportunity, and it maps directly to Requirements #2 and #4.

## 3. Existing vocabularies (what we have to seed Requirement #3)

**Topics seen in whole corpus:** just `Investment Arbitration` (1 occurrence). Effectively no existing topic taxonomy to extend — you will be **building** it, not augmenting it.

**Keywords seen (37 unique across 5 docs):** fragmented, mixed casing, inconsistent granularity:
- Structural/conceptual: `Arbitration`, `Investment Arbitration`, `International Commercial Arbitration`, `state immunity`, `Generative AI`
- Instrument names: `ICSID Convention`, `New York Convention`
- Case names: `Infrastructure Services Luxembourg`
- Phenomena: `Arbitrator Bias`, `Hallucination`, `Blackbox`, `Issue Conflict`
- Lowercase `arbitration` AND Capitalized `Arbitration` both appear → **normalization is a prerequisite for any dictionary**

**Organizations (3):** ICSID, Silicon Valley Arbitration and Mediation Centre, Chartered Institute of Arbitrators — this is a natural seed list for an organization NER model.

**Countries (5 tags across docs):** Latvia (×2 variations), + others. Attribute `value="LV"` is already ISO-2 coded — use that as the canonical key.

## 4. Structural signals that already exist for "relationships" (Requirement #3)

### Cross-reference graph (`<xref>`)

- **51 internal edges** in the sample (after dropping `mailto:` / `http`)
- **Edge types:** `ref` (37, footnote citations to case law / other docs), `sec` (9, internal section jumps), `toc` (5, TOC → chapter)
- **Only 5/51 resolve within the sample corpus** — because the sample is small. At full-corpus scale this graph will be dense and is the single most valuable "free" relationship signal already in the data.

**Practical use:** the xref graph gives you document-to-document relations without any NLP. Combine with keyword co-occurrence (below) for keyword-to-keyword edges.

### Keyword co-occurrence (within same doc)
Already yields clean clusters even from 5 enriched docs, e.g.:
```
ICSID Convention  <->  state immunity
ICSID Convention  <->  New York Convention
Arbitration  <->  Artificial Intelligence
Arbitration  <->  Hallucination
```
At scale this becomes the backbone of the topic/keyword dictionary.

### Structured facets already in commentary (`<juris-description>`)
Case-notes carry **parties, court, date, case name** as first-class fields — e.g. `KLI-JOIA-420502` parses into `DJP and others v. DJO [2024] SGHC(I) 24, 2025-04-08, parties=[DJP, DJO,...]`. For Requirement #4 (awards / case decisions ingestion), **this is the schema you want to target**: structured case fields + NLP-generated topics/keywords on top.

## 5. Body length & chunking implications (Requirement #4)

Across the 20 docs with real body text:

| Metric | Words |
|---|---|
| min | 575 |
| p25 | 7,101 |
| median | 11,798 |
| p75 | 14,823 |
| **max** | **40,580** (Austria ICCA Handbook report) |
| mean | 12,358 |

- **Median ~12k words ≈ 15–25k tokens.** Most docs comfortably fit in a modern LLM context window for single-pass summarization/tagging; a minority (top 10%) will need chunked passes.
- Every doc is cleanly sectioned (`<section id="S0001">…`), so chunking should respect section boundaries — the parser already exposes `Section.id/number/title/text` for this.

### Three "empty body" placeholders — **keep an eye on these at full-corpus scale**
- `KLI-KA-AI-Arbitration-2026-b001.xml` (front matter / blurb)
- `KLI-KA-AI-Arbitration-2026-Ch01.xml` (`<text><p/></text>` — genuine placeholder, chapter not yet in sample)
- `TOC-AI-Arbitration-2026.xml` (TOC has no body by design)

These are true-negatives, not parse failures.

## 6. How each requirement maps to what we now know

### Req #1 — Blog → paid-content bridging
- **What's here:** titles, abstracts (first `<abstract>` block inside `<text>`), keywords for ~22% of docs, container hierarchies (journal / book / chapter).
- **What's missing:** consistent topic/keyword layer to match blog topics against.
- **Action:** build the keyword/topic vocabulary from Req #3 first; match blog content into it via embeddings; rank paid docs by `(keyword overlap × recency × cust_group relevance)`.

### Req #2 — Enrich KA / KCL / KIPL with new tags & hooks
- **What's here:** 95% of docs have no topics; 78% have no keywords. Clean sectioned text. Container metadata (journal/book) gives a weak per-doc domain prior.
- **Action:**
  - Topic classification (zero-shot LLM or fine-tuned classifier) at **document** and **section** level.
  - Keyword extraction (YAKE/KeyBERT + LLM curation) normalized against the seed 37-keyword vocab.
  - **Write the results back into `<text-description>`** — the DTD already has the slots waiting.

### Req #3 — Topic/keyword dictionary with relationships
- **What's here:** `<xref>` graph (51 edges in sample), keyword co-occurrence patterns, `<cust-group>` product clusters.
- **What's missing:** a controlled vocabulary; the 37 keywords already show casing/inflection duplicates.
- **Action:**
  1. Build the vocabulary (nodes).
  2. Build the edges in parallel from three sources: **xref graph**, **intra-doc co-occurrence**, **embedding similarity** between keyword phrases.
  3. Promote stable clusters to `<topic>` candidates.

### Req #4 — Ingest primary content (awards, case decisions)
- **What's here:** `<commentary>` + `<juris-description>` already models parties, court, date, case name, case number. That's the target schema.
- **Action:**
  - New ingestion pipeline emits the same structure: parties/court/date (deterministic extraction from award PDFs), then topics/keywords from the same NLP stack as Req #2.
  - Link into the dictionary built in Req #3 for immediate search/assistant integration.

## 8. Schema & parsing reference (Data-Engineer view)

This section answers four specific questions a data engineer typically asks before committing to a pipeline: **(a) what is the XML hierarchy, (b) which tags are payload vs metadata, (c) what data-quality issues exist, (d) how hard is it to ship as LLM-ready JSON with semantic chunking.**

### 8a. XML hierarchy & schema

All files: DTD `-//KLI//DTD KEA-BASIC//EN` → `keabasic.dtd` (external). The DTD is external to the sample, so the schema below is **reconstructed from the 23 KA files** (see §8c for the subset caveat).

```
<KEA-publ>
└── <essay | legislation | commentary | caselaw | booktoc>   ← 5 doc types
    ├── <frontmatter>                                        ← METADATA (see 8b)
    │   ├── <title>
    │   ├── <authorgroup><author><name>First<surname value="123">Last</surname></name></author>…
    │   └── <publication-info publication-type="journal|book|loose-leaf">
    │       ├── <title>       # container (journal / book) title
    │       ├── <short-title/>
    │       ├── <authorgroup><editor>…</editor></authorgroup>
    │       ├── <publ-cluster/>
    │       ├── <edition><volume/><issue/><supplement/></edition>
    │       ├── <publ-date value="YYYYMMDD"/>
    │       ├── <publisher id="publ-020"><name>…</name></publisher>
    │       ├── <isbn/> <issn/>
    │       ├── <range first="533" last="566" type="pp"/>
    │       ├── <cust-group value="KA|IHB|CH"/>              # may repeat
    │       └── <copyright>
    │
    ├── <text-description>                                   ← EXISTING ENRICHMENT (see 8b)
    │   ├── <topic id="Arb-001">Investment Arbitration</topic>     # 0..n, mostly empty
    │   ├── <keywords><keyword>…</keyword>…</keywords>             # 0..n
    │   └── <text-scope>
    │       ├── <country value="LV">Latvia</country>               # 0..n
    │       └── <organization value="ORG0244">ICSID</organization> # 0..n
    │
    └── <text>                                               ← PRIMARY PAYLOAD (essay)
        ├── <abstract><p>…</p></abstract>                   # optional
        └── <section id="S0001">                            # nests up to depth 3
            ├── <number/> <title/>
            ├── <p>… <it/> <bd/> <sc/> <xref value="…" type="url"/> <note id="n1"><p/></note> …</p>
            ├── <blockquote><p/></blockquote>
            ├── <list type="digit|ndash|bullet"><item><number/><p/></item>…</list>
            ├── <?page nr="533"?>                            # PI, not an element
            └── <section>…</section>                         # recursive
```

**Doc-type-specific body containers:**
| Doc type | Body wrapper | Body element |
|---|---|---|
| `essay` | `<text>` | `<section>` |
| `legislation` | `<legis>` → `<legis-text>` | `<legis-section>` |
| `commentary` (case-note) | `<juris-comment>` | `<section>` + `<juris-description>` |
| `commentary` (legis-note) | `<legis-comment>` | `<section>` + `<legis-description>` |
| `booktoc` | `<booktoc-text>` | `<booktoc-section>` (navigation only) |

### 8b. Payload vs metadata tag classification

Five tag classes, each handled differently by the pipeline:

| Class | Tags | Role | How pipeline treats them |
|---|---|---|---|
| **Structural identity** | `<KEA-publ>`, `<essay>`, `<legislation>`, `<commentary>`, `<caselaw>`, `<booktoc>` (+ `id`, `lang`, `essay-type`, `commentary-type` attrs) | Routing / schema dispatch | Selects the parse path; emitted as top-level JSON field `doc_type`. |
| **Bibliographic metadata** | `<frontmatter>`, `<title>`, `<authorgroup>/<author>/<name>/<surname>`, `<editor>`, `<publication-info>`, `<publ-date>`, `<publisher>`, `<isbn>`, `<issn>`, `<edition>/<volume>/<issue>/<supplement>`, `<range>`, `<cust-group>`, `<copyright>`, `<updated>` | Catalog facets, filters, access control | Extracted to flat JSON keys. Never included in LLM chunk text. |
| **Existing enrichment (metadata)** | `<text-description>`, `<topic>`, `<keywords>/<keyword>`, `<text-scope>/<country>`, `<text-scope>/<organization>` | Retrieval tags + seed taxonomy | Extracted to flat JSON lists. Also passed to LLM as prior tags for few-shot enrichment. |
| **Case-specific metadata** (commentary/caselaw) | `<juris-description>`, `<juris-parties>/<juris-party>`, `<juris-court>`, `<juris-authority>`, `<juris-date>`, `<juris-name>`, `<juris-number>`, `<references>` | Structured case facets | Separate JSON block; exposes deterministic retrieval facets (court = X, date range Y..Z). |
| **PRIMARY TEXT PAYLOAD** | `<text>`, `<legis-text>`, `<juris-text>`, `<juris-comment>`, `<legis-comment>` + descendants: `<section>`, `<legis-section>`, `<p>`, `<abstract>`, `<blockquote>`, `<list>/<item>`, inline `<it>`, `<bd>`, `<sc>`, `<number>` | **The content** | Recursively flattened to plain text; inline markup dropped; `<xref>` captured separately as edges; `<note>` footnotes captured but extractable on demand (not mixed inline). |

**Critically:** `<xref value="…" type="url|sec|toc">` straddles the line — it appears **inside** payload text but **is** metadata (a cross-reference edge). The parser's decision: extract it to `doc.xrefs` as graph edges and drop it from the plain-text flattening. Otherwise retrieval would see URLs and cryptic IDs inside prose.

### 8c. Data-quality issues (measured, not hand-waved)

| # | Issue | Quantified in KA sample | Impact | Pipeline response |
|---|---|---|---|---|
| **1** | **Commented-out legacy `<frontmatter>`** kept alongside the current one | **13/23 files (56%)** | Naïve XPath like `//frontmatter` returns both and duplicates authors/dates | Strip XML comments *before* parsing — parser does this |
| **2** | **Internal DOCTYPE entity subsets** (`<!DOCTYPE … [ <!ENTITY ... NDATA GIF> … ]>`) declaring binary GIF figures | **1/23 files** (`KLI-KA-AI-Arbitration-2026-Ch04.xml`) | Default stdlib ET rejects the file entirely | Strip the DOCTYPE (incl. `[...]` subset) before parse — parser does this |
| **3** | **External DTD reference** (`keabasic.dtd`) not shipped with samples | All 23 files | Any validating parser / XSD-style tool will fail; also blocks canonical element whitelisting | Use non-validating parse; reconstruct schema from data |
| **4** | **Casing / inflection duplicates in keyword vocab** | `Arbitration` AND `arbitration` both appear; overall 37 keywords from 5 docs with `2` case-collision | Keyword dictionary (Req #3) builds broken nodes | Normalize: casefold + lemmatize + dedupe before emitting |
| **5** | **Empty-body placeholder files** (author-review stubs, TOCs) | **3/23 files** — `KLI-KA-AI-Arbitration-2026-Ch01.xml`, `-b001.xml`, `TOC-AI-Arbitration-2026.xml` | Chunking receives 0 chunks; must not crash downstream | Parser emits `word_count == 0`; pipeline should filter, not fail |
| **6** | **Missing `<topic>` and `<keywords>`** in the large majority of docs | 95.7% have no topic; 78.3% have no keywords | Search/assistant surfaces have almost nothing to match on today | This is the work in Req #2 |
| **7** | **`lang` attribute missing** on `booktoc` | 1/23 files | Language-based routing can misclassify | Default to container's language when missing |
| **8** | **Duplicate `<topic>` id values unobserved** (good news) | 0 collisions | — | Safe to use `topic/@id` as canonical key at full corpus scale |
| **9** | **Missing authors on book-front-matter files** | `KLI-KA-AI-Arbitration-2026-b001.xml` | Author indices / citations break silently | Allow null authors only for `*-b001`-style front-matter; flag elsewhere |
| **10** | **Section nesting up to depth 3** + inline `<note>` (footnotes) containing their own `<p>`/`<xref>` | `section depth ≤ 3` in 26% of KA files; 11/23 have `<abstract>` | Flat "take top-level sections" chunking loses sub-structure; footnotes pollute inline text | Parser emits top-level sections for EDA; for retrieval chunking, recurse to leaf `<section>`/`<p>` and extract `<note>` separately |
| **11** | **Page markers as processing instructions** (`<?page nr="533"?>`) inline in text | Every paginated doc | ET exposes PIs separately; naive `.text`/`.tail` concatenation can duplicate surrounding whitespace | Skip PIs during text flattening (parser already does) |

### 8d. Ease of parsing to LLM-ready JSON + chunking plan

**Verdict: Low–medium complexity. After the 5 pre-processing fixes above, the hierarchy maps cleanly to JSON.** Every KA sample file parses with ~200 lines of stdlib Python (`src/ka_parser/parser.py`), no lxml, no XSD.

**Target JSON shape (what `parsed_docs.json` already emits):**
```json

  "doc_id": "KLI-JOIA-420501",
  "doc_type": "essay",
  "essay_type": "articles",
  "lang": "en",
  "metadata": {
    "title": "State Immunity and the Recognition…",
    "authors": ["Darren Leow"],
    "editors": ["Rémy Gerbay"],
    "container_title": "Journal of International Arbitration",
    "publisher": "Kluwer Law International",
    "publ_year": 2025, "publ_date": "20251000",
    "isbn": null, "issn": "0255-8106",
    "volume": "42", "issue": "5",
    "pages": {"first": "533", "last": "566"},
    "cust_groups": ["KA", "CH"],
    "copyright": "..."
  },
  "enrichment": {
    "topics": ["Investment Arbitration"],
    "keywords": ["ICSID Convention", "state immunity", ...],
    "countries": [],
    "organizations": ["International Centre for Settlement of Investment Disputes"]
  },
  "case_metadata": null,            // populated only for commentary/caselaw
  "body": {
    "word_count": 14511,
    "sections": [
      {"id": "S0001", "number": "1", "title": "INTRODUCTION", "text": "…", "word_count": 1240},
```

**Semantic chunking recommendation (tuned to measured section-size distribution):**

KA top-level sections span **9 to 9,911 words** (median ≈ 950). A single global chunk size will fail at both ends. Recommended policy:

| Scenario | Chunk strategy |
|---|---|
| Section ≤ 800 words | **Whole section = 1 chunk.** Carry forward: `doc_id`, `section.id`, `section.title`, `doc.metadata`, `doc.enrichment`. |
| Section 800–2,500 words | **Split on paragraph boundaries** within the section, target ~500 words/chunk with 50-word overlap. Preserve section breadcrumbs. |
| Section > 2,500 words | **Recursive split on nested `<section>`** (depth ≤ 3 in KA) before falling back to paragraph splitting. |
| Any size | Emit `<abstract>` (when present — 11/23 KA files) as a dedicated chunk tagged `chunk_type: "abstract"` — it is a ready-made document summary. |
| Footnotes | Extract `<note>` subtrees to a sibling list (`"footnotes": [...]`), do **not** inline them into section text — they destroy semantic coherence for retrieval. |

**Per-chunk metadata to attach** (for retrieval filtering / citation):
```
doc_id, doc_type, chunk_id, section_path, section_title,
publ_year, container_title, cust_groups, lang,
topics, keywords, countries,
source_byte_span   // useful for UI highlight-on-source
```

**What is easy:** flat metadata extraction, section walking, xref edge collection, emitting JSON.

**What needs care (landmines):**
1. Legacy commented-out frontmatter → 56% of KA files would duplicate metadata without comment-stripping.
2. Internal DOCTYPE entity subsets → 1/23 files fails a stock parser; not optional to handle.
3. Footnotes (`<note>`) live *inside* `<p>` — naïve `.itertext()` concatenates them mid-sentence, garbling retrieval.
4. Inline `<xref>` URLs would leak into the LLM context if not separately extracted.
5. Three doc types (`essay` / `legislation` / `commentary`) have **different** body wrappers (`<text>` vs `<legis-text>` vs `<juris-comment>`/`<legis-comment>`); a single XPath like `//text` misses most content.

These are all handled in the current parser. A fresh implementation that ignored them would produce silently-wrong output, not crash — which is the worst failure mode for an LLM pipeline.

## 9. Recommended next EDA passes (once more data lands)

1. **Section-level length distribution** — drives chunk-size defaults for retrieval.
2. **Abstract extraction** — first `<abstract>` in essays is a dense document-summary candidate; worth surfacing as its own column.
3. **Author collaboration graph** — present infrastructure already collects authors/editors; at scale this segments "who writes about what" and supports author-based recommendation.
4. **xref target resolution at full-corpus scale** — today 5/51 resolve; at full scale this is the free relationship graph.
5. **Footnote / citation parsing** — `<note>` blocks contain rich case-law citations that, when parsed, feed back into the dictionary.
6. **Cross-DTD mapping** — repeat this exercise against `KCL-xml-samples/` to confirm schema parity before committing to a unified enrichment pipeline.
